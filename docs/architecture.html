01 The insight that sets every priority

Crypto exchange round-trip latency is milliseconds (≈5–10 ms), not microseconds — the feed arrives over TCP from a cloud endpoint. Shaving your own code from 50 µs to 10 µs is invisible against a 5,000 µs exchange floor. That doesn't mean "don't build for low latency." It means the project has two clean goals, and we stay honest about which is which.

Two goals, mutually reinforcing
Nanosecond craft — ring buffers, order book, cache discipline, lock-free hand-offs, real latency measurement — is the learning goal, built to production standard because that's the point.

What actually determines crypto PnL is correctness of order-book state — above all sequence-gap detection & recovery. Crypto feeds drop messages without disconnecting: the sequence ticks forward, the stream stays live, but you silently missed an update and your book is now wrong. That's a first-class module, not a footnote.

02 Threading — one hot thread, offload the cold path

Crossing cores costs a cache-coherency round-trip (~40–100 ns per hop, plus jitter). Thread-per-stage is a throughput optimization that raises per-message latency — the opposite of what the critical path wants. So the hot path is a single thread running parse → book → strategy → risk → sim as plain function calls. Threads appear only where work can't share the hot thread: blocking I/O and slow/cold work, connected by lock-free SPSC ring buffers.

Why the cold path gets the queue and the hot path doesn't

Ring buffers (SPSC, lock-free, cache-line padded) connect I/O↔hot and hot↔cold — reader→hot, hot→sender, hot→logger, feed→recovery. That's where you build and learn the lock-free primitive, because that's where it belongs. You do not sprinkle queues between hot stages, where they'd only add latency. Being able to explain this is the interview payoff.

03 Agnostic core, venue-specific edges

The normalized BookDelta is already venue-neutral — SymbolId, integer-tick price_t, qty_t, Side. The feed handler is the only thing that knows Coinbase; swap in a CME feed handler that emits the same BookDelta and the entire core is unchanged. SymbolTable already carries per-symbol tick / lot / scale — the exact generalization point. Don't build "a crypto system." Build a trading core with a crypto adapter bolted on.

04 Realism & phasing

Phase 1 · ship it

Get it running — the resume artifact

  • Single hot thread; agnostic core
  • Real order book with sequence-gap recovery
  • Real latency taps (platform::now_ns())
  • Core pinning where the OS allows (Linux)
  • Cache-line discipline in the ring buffer
  • Full loop → simulated fills + PnL
Phase 2 · pure upside

"I understand the real stack"

  • Mock UDP multicast feed on localhost
  • Busy-polled, hardware-timestamped
  • Measured against the WebSocket path
  • Write-up: where kernel bypass would and wouldn't help — and why not here

Why phase it this way: knowing why not to use something, and proving you understand it, is a stronger signal than a half-broken DPDK integration. Resume line — "characterized the latency budget and documented where kernel-bypass would and wouldn't help."
